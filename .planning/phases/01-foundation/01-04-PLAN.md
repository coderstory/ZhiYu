<objective>
Wire the splash screen and main activity entry point: AndroidX SplashScreen API with DataStore keep condition, edge-to-edge rendering, and a post-splash foundation placeholder composable.

**Purpose:** The entry point for the entire app. Splash screen avoids white flash on MIUI (PITFALL-7). DataStore keep condition ensures preferences are loaded before the user sees content. Edge-to-edge is mandatory on Android 16 (API 36).

**Output:** MainActivity.kt (splash + edge-to-edge + theme wiring + placeholder), FoundationPlaceholder.kt (status card composable).

**Critical execution order (PITFALL-7 mitigation):**
1. `installSplashScreen()` — MUST be first
2. `enableEdgeToEdge()` — MUST be before super.onCreate()
3. `super.onCreate()` — MUST be after both
4. `setKeepOnScreenCondition { !isDataStoreReady }` — keeps splash visible until DataStore emits
5. `setContent { ZhiYuTheme { FoundationPhasePlaceholder() } }` — app content
</objective>

  <action>
    Create the single-Activity entry point with the correct initialization order.

    **1. `MainActivity.kt`**:
    - Package: com.zhiyu.app
    - class MainActivity : ComponentActivity()

    - override fun onCreate(savedInstanceState: Bundle?):
      - // 1st: Install splash screen BEFORE super.onCreate (D-10, PITFALL-7 mitigation #1)
      - `val splashScreen = installSplashScreen()`

      - // 2nd: Enable edge-to-edge BEFORE super.onCreate (Android 16 mandatory, PITFALL-7 mitigation #2)
      - `enableEdgeToEdge()`

      - // 3rd: super.onCreate AFTER both (PITFALL-7 mitigation #3)
      - `super.onCreate(savedInstanceState)`

      - // 4th: Set up DataStore keep condition (D-09: splash stays until DataStore ready)
      - `var isDataStoreReady by mutableStateOf(false)`
      - `splashScreen.setKeepOnScreenCondition { !isDataStoreReady }`

      - // 5th: Set Compose content
      - `setContent {`
        - Obtain AppPreferences from Koin:
          - `val appPreferences: AppPreferences = org.koin.android.ext.android.get(this)` or use `org.koin.compose.koinInject()`
          - Using koinInject is the Compose-idiomatic approach:
            - `val appPreferences: AppPreferences = koinInject()`
        - Collect themeMode with initial value:
          - `val themeMode by appPreferences.themeMode.collectAsState(initial = ThemeMode.SYSTEM)`
        - Mark DataStore ready after first emission:
          - `LaunchedEffect(themeMode) { isDataStoreReady = true }`
        - Apply theme wrapper:
          - `ZhiYuTheme(themeMode = themeMode) {`
            - `FoundationPhasePlaceholder()`
            - `}`
      - `}`

    - Import notes:
      - `androidx.activity.enableEdgeToEdge` — NOT a typo, it's in `androidx.activity:activity-ktx` which is transitive via Compose
      - `androidx.activity.installSplashScreen` — from core-splashscreen compat
      - `org.koin.compose.koinInject` — from koin-androidx-compose
      - `androidx.compose.runtime.LaunchedEffect` — to set isDataStoreReady
      - `androidx.compose.runtime.collectAsState` — to observe DataStore flow
      - `com.zhiyu.app.model.ThemeMode` — for initial value
      - `com.zhiyu.app.ui.theme.ZhiYuTheme` — theme wrapper from Plan 3
      - `com.zhiyu.app.ui.components.FoundationPhasePlaceholder` — from Plan 4 Task 2

    - Critical sequence (PITFALL-7):
      ```kotlin
      val splashScreen = installSplashScreen()  // 1
      enableEdgeToEdge()                         // 2
      super.onCreate(savedInstanceState)          // 3
      ```

      Wrong order (e.g., super.onCreate before installSplashScreen) causes white flash on MIUI devices because the splash system is initialized after the activity has already started rendering. The order above MUST be strict.

    - DataStore first-read blocking (PITFALL-8):
      - `appPreferences.themeMode` has `.flowOn(Dispatchers.IO)` from Plan 2, so the first read is on IO dispatcher.
      - However, `collectAsState` needs an initial value. Use `collectAsState(initial = ThemeMode.SYSTEM)` to immediately provide a value while the IO read happens.
      - `setKeepOnScreenCondition { !isDataStoreReady }` means splash stays visible until DataStore actually emits (confirming preferences are loaded).
      - The LaunchedEffect sets isDataStoreReady = true after the first emission, which immediately triggers splash dismissal.

    - No setContentView, no XML layout — Compose-only activity.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SplashScreen API -> Activity.onCreate | The system splash renders before the Activity has initialized. Initialization order is critical. |
| DataStore file read -> Main thread | Blocking read from DataStore preferences could delay first frame |
| App content -> system bars | Edge-to-edge rendering means content draws behind system bars; insets must be applied |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-FND-10 | Denial of Service | installSplashScreen order | mitigate | Enforce strict ordering: installSplashScreen -> enableEdgeToEdge -> super.onCreate (PITFALL-7). Documented in code comments. |
| T-FND-11 | Denial of Service | DataStore first-read blocking | mitigate | `.flowOn(Dispatchers.IO)` on all DataStore flows (Plan 2). `setKeepOnScreenCondition` holds splash until first emission. `collectAsState(initial = ...)` provides immediate default. |
| T-FND-12 | Information Disclosure | White flash on MIUI | mitigate | Solid #FF6B35 splash background (no complex drawables), `postSplashScreenTheme` set to Theme.ZhiYu, correct initialization ordering. |
</threat_model>
